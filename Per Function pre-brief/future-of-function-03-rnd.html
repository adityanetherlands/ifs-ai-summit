Future of Function · 03

Future of R&D

What happens when AI agents write more code than your engineers - and you have to rethink the entire role of a programmer?

1. The Shift

GitHub Copilot users generate code 55% faster than those without AI assistance. At GitHub’s own scale, that’s equivalent to adding 880 additional engineers without hiring. But the real shock isn’t speed - it’s that the best engineering teams in 2025 aren’t removing engineers. They’re forcing their engineers to become architects and decision-makers instead of typists.

Amazon’s engineers using CodeWhisperer report that AI generates 40-60% of their production code. Quality doesn’t drop. Defect rates don’t spike. The code gets better because engineers spend 40% less time on boilerplate and 60% more time on design, architecture, and solving novel problems. One Amazon team shipped a feature in three weeks that would have taken six months hand-written.

Google DeepMind’s AlphaCode solved programming problems at the level of a top 54% competitive programmer. Outperforming 85% of humans on unseen problems. It doesn’t replace engineers. It surfaces which engineers can solve problems that machines cannot.

Cursor, the AI-native code editor, has 500,000+ daily active users. Teams using Cursor report 50% reduction in time spent on routine refactoring and legacy code maintenance. The work that strangles productivity at scale. That 50% time isn’t being lost. It’s being redirected to building new systems, not maintaining old ones.

Three companies, three strategies, one pattern: the engineering function is being fundamentally restructured. Not by automating engineers away, but by collapsing the mechanical translation between thought and code, forcing humans to think differently about what engineering actually is.

2. Your Pre-Summit Homework

Three things to bring to the session. Not to present, but to think about. Write them down. Short answers are fine. What matters is that you’ve actually wrestled with them.

1. A bold vision for 1 January 2027

If you fast-forwarded 8 months, what does your R&D function look like? Not incremental. Drastically different. What’s disappeared? What’s new? What does “an engineering team” even mean? Write the headline version: if someone wrote a profile of your function in January 2027, what would it say?

2. The case for transformation

Why does your R&D function need to transform, and why now? Not “AI is important” generically. The specific competitive, technical, or structural reason transformation is non-negotiable for you. What breaks if you stay as you are? If you can’t articulate this in two sentences, your team won’t follow you.

3. The KPIs that matter in the new era

What are the 2-3 metrics that tell you R&D is winning in an AI-native world? Cycle time? Velocity? Bug escape rate? Feature adoption? Developer productivity? Something entirely new? The old KPIs of lines written and hours billed measure activity. What measures outcome in a world where boilerplate is automated?

3. Directional Nudges - Sense / Decide / Execute

Sense: From Code Review to Codebase Intelligence

What if every engineer had an always-on AI agent that understood your entire codebase, surfaced bugs before they shipped, and predicted architectural debt three sprints ahead?

This is already real. Tools like Sourcegraph Cody understand 10+ million lines of code in context. Snyk continuously scans repositories and finds vulnerabilities before humans review them. Linear’s AI integrates with your codebase and surfaces patterns: which code is changing most? Which modules are brittle? Where is complexity concentrated? Where should you refactor?

The structural change: the “sensing” layer stops being manual code review and starts being continuous codebase intelligence. Humans stop reviewing code for bugs and start reviewing AI-flagged anomalies for judgment calls. The volume of code auditable by a human team goes from 2-3 reviews-per-person-per-day to 50+ anomalies-reviewed-with-context.

Decide: From Architecture Reviews to AI-Informed Tech Choices

What if every architectural decision - microservices vs. monolith, build vs. buy, database selection, API design - came with 20+ pre-evaluated alternatives, with trade-offs, before the meeting started?

Claude and similar reasoning models can synthesise the entire design decision landscape. Feed it your system constraints (latency, data volume, cost), and it surfaces not just the best choice but the second and third best, with clear reasoning on when each wins. Git and architectural decision records (ADRs) are now searchable context. The AI can reference every previous decision to keep new ones consistent.

The structural change: the decision cycle compresses from weeks of whiteboarding to hours. Not because decisions are worse, but because the bottleneck was never judgment. It was information preparation. When an AI can prepare 20 architectures faster than humans can prepare one, the only thing left is the debate. And that’s what architects should have been doing all along.

Execute: From Manual Testing to AI-Generated Test Suites

What if comprehensive test coverage - unit tests, integration tests, edge cases, regression suites - was generated automatically and maintained continuously as code changes?

QodoAI, Codium, and similar tools generate test suites by analysing code. They don’t replace QA. They collapse the time between code generation and test coverage from weeks to minutes. Defect escape rate doesn’t climb. It drops. Because AI tests for branches humans forget to check.

The structural change: execution visibility becomes continuous. The release pipeline doesn’t pause for manual testing. It runs automatically, with AI-generated and maintained test suites catching regressions before humans notice them. QA shifts from writing tests to validating edge cases and user experience. The work machines still can’t do well.

The Loop Closes

In the AI-native model, engineering stops operating on quarterly sprints. Sensing is continuous. Deciding is rapid (days, not weeks). Execution is tracked and tested autonomously. The engineering standup shifts from “what did we ship” to “what should we prioritise”. Because the information supply chain is instant.

Thought Exercise

If 50% of your engineering time is freed by AI handling boilerplate and testing, what would your team actually build with that capacity? Not more features - what capability gap would you finally address? If the answer is “we have no idea,” that’s the strategic conversation.

5. Hands-On Exercises

Do at least one before the session. These are designed to give you first-hand experience of what AI-native engineering feels like - not just an opinion about it.

Exercise 1: The AI Pair Programming Session  ·  30 min

Pick a real bug or feature in your codebase. Spend 15 minutes solving it the traditional way (writing code, debugging, testing). Then spend 15 minutes using Cursor or GitHub Copilot with full codebase context and ask the AI to solve the same problem. Compare: How long did each take? How different is the AI solution? What did the AI do better? What did it miss?

Output: A side-by-side comparison of hand-coded versus AI-generated solution with time measurements.

Tools: Cursor or GitHub Copilot, your codebase

Exercise 2: The AI Test Suite Generator  ·  60 min

Pick a module in your codebase with incomplete test coverage or known quality issues. Feed the source code into QodoAI or Claude and ask it to generate a comprehensive test suite: unit tests, integration tests, edge cases. Run the generated tests against your code. Measure: How many pass? How many catch real bugs? How does coverage compare to your existing manual tests?

Output: An AI-generated test suite with coverage metrics and comparison to existing tests.

Tools: QodoAI or Claude, your codebase

Exercise 3: The Architecture Decision Engine  ·  90 min

Take a real technical decision your team is debating (microservices vs. monolith, database migration, API redesign). Write a 1-2 paragraph problem statement with constraints (latency, cost, team size). Feed it to Claude with your current codebase context and ask it to generate 5+ architectural alternatives with trade-off analysis. Build an interactive HTML report showing outcomes, risks, and when each alternative wins. Use this as a prototype for how architecture decisions could be made in the future.

Output: An interactive architecture decision report that engineers could use to evaluate trade-offs.

Tools: Claude, Python or JavaScript, HTML/CSS
